## Section 1: Onboarding Pipeline — End to End

### 1.1 Complete step-by-step process to add a new capability

| Step | Action | Time | Files/Commands |
|------|--------|------|----------------|
| 1 | Write executor | 30-120min | `src/capabilities/{slug}.ts` with `registerCapability()` |
| 2 | Auto-registered | 0min | `auto-register.ts` discovers at startup — no manual import |
| 3 | Create YAML manifest | 10-30min | `manifests/{slug}.yaml` with 12+ required fields |
| 4 | Run pipeline | 2-5min | `npx tsx scripts/onboard.ts --discover --manifest ...` |
| 5 | Review auto-generated fields | 5min | Check `expected_fields` and `output_field_reliability` in manifest |
| 6 | Run validation | 1min | `npx tsx scripts/validate-capability.ts --slug {slug}` |
| 7 | Run smoke test | 1-2min | `npx tsx scripts/smoke-test.ts --slug {slug}` |
| **Total** | | **~50-160min** | (executor complexity dominates) |

**Auto-generated by the pipeline (--discover mode):**
- `expected_fields` — derived from live execution output
- `output_field_reliability` — all fields initially marked `guaranteed`
- 5 test suites: known_answer, schema_check, negative, edge_case, dependency_health
- Known-answer test validation rules (from manifest fixtures)
- Baseline output and baseline hash

**Manually provided (irreducible):**
- The executor code itself
- `health_check_input` — a real input that always works
- `known_answer` test input — a known-good entity
- Field reliability refinements (demote `guaranteed` → `common`/`rare` as needed)
- At least 1 limitation (honest coverage assessment)
- Pricing, category, data source metadata

### 1.3 Incomplete onboarding failure mode

**What happens if someone creates the executor but forgets the pipeline:**

1. **Executor auto-registers** at startup via `auto-register.ts` — the capability code is loaded
2. **No DB row** → `matchCapability()` can't find it → customers can't call it via `/v1/do`
3. **No test suites** → no SQS score → no quality data
4. **Readiness check** returns `ready: false` with issues: "No database row", "No test suites", etc.
5. **Lifecycle state** never enters `active` — stays in non-existent/draft

**Verdict:** An untested capability CANNOT serve customer traffic. The matching layer requires `is_active = true` + `lifecycle_state IN ('active', 'degraded', 'probation')` + `visible = true` (for task matching). A capability that hasn't been through the pipeline will have `lifecycle_state = 'draft'` and `visible = false`.

**However:** Direct slug lookup allows `probation` state. If someone manually sets `lifecycle_state = 'probation'` and `is_active = true` in the DB, a capability could serve traffic with zero tests and zero SQS. The onboarding pipeline is enforced by convention, not by database constraints.

The pipeline runs these validations (`onboard.ts` lines 112-166, `validate-capability.ts` 79-256):

| Check | Automated? | Blocks deployment? |
|-------|-----------|-------------------|

### 1.5 Bulk onboarding failure modes

At 100 simultaneous capabilities:

| Risk | Impact | Mitigation |
|------|--------|-----------|
| API rate limits (upstream) | `--discover` calls each API once | Pipeline is sequential — no parallel discovery |
| DB connection exhaustion | 100 INSERT sequences | Pipeline runs serially per manifest — not parallelized |
| Browserless quota | 100 scraping probes | ~$0.50 total at Browserless.io rates |
| LLM API cost (Claude for AI caps) | ~$0.10-0.50 per AI capability discovery | Budget ~$50 for 100 caps |
| Time | ~5min per cap sequential | ~8 hours sequential; parallelizable to ~30min |

**Bottleneck at scale:** The pipeline is sequential per manifest file. No batch mode exists. Adding 1,000 capabilities requires running the script 1,000 times. A batch pipeline would be needed.

### 1.6 Output field reliability population

Three paths to populate `output_field_reliability`:

1. **Manual** — Author writes reliability levels in the manifest YAML
2. **Auto-discover** (`--discover` flag) — Pipeline executes the capability, inspects the output, and marks ALL fields as `guaranteed`. Human then demotes to `common`/`rare` as needed.
3. **Backfill** (`--backfill --discover`) — Re-runs discovery on existing capabilities and updates manifest + DB

**Key design:** Only `guaranteed` fields get `not_null` test assertions. This prevents the "expected non-null on optional field" problem that broke 8 EU registries early on.

### 1.7 Readiness gates

```
                          ┌──────────────────────────────────────┐
                          │   Can serve customer traffic via      │
                          │   POST /v1/do?                       │
                          └──────────────────────────────────────┘
                                          │
                    ┌───────────────────────────────────────┐
                    │   matchCapability() requires:          │
                    │   • is_active = true                   │
                    │   • lifecycle_state IN                  │
                    │     (active, degraded, probation)       │
                    │   • visible = true (task matching)      │
                    │   • price ≤ max_price_cents             │
                    │   Optional: min_sqs quality gate        │
                    └───────────────────────────────────────┘
```

**A capability with NO SQS score can serve traffic** if it reaches `probation` state. The `min_sqs` parameter is caller-specified, not platform-enforced. There is no platform-wide SQS floor on execution (the platform floor of 25 from the SQS Constitution applies to display/ranking only).

### 1.8 Capability dependencies

Solution steps define multi-capability pipelines (`solution_steps` table with `step_order`, `can_parallel`, `parallel_group`). But for individual capability dependencies:

- **No formal dependency declaration** between capabilities
- **No runtime dependency graph** — each capability is independently executed
- **DataProvider fallback chains** (`data-provider.ts`) provide multi-source resilience within a single capability
- **Solution-level health** is computed as weighted average of step SQS scores (floor-aware: min+20 cap)

## Section 2: Testing System — Mechanics and Coverage

### 2.1 Test type inventory

| Type | What it tests | Input | Pass/fail | Cost per run | Count | Example pass | Example fail |
|------|--------------|-------|-----------|-------------|-------|-------------|-------------|
| **known_answer** | Correctness — real API returns expected data | From manifest (real entity) | Validation rules on output fields (operators: not_null, equals, contains, etc.) | Real API call (~€0.01-0.50) | ~260 | `iban-validate` returns `{valid: true}` for known valid IBAN | Field `valid` is null |
| **schema_check** | Schema structural integrity | Dry-run — NO external call | Input schema has required+properties, output schema valid, executor exists | FREE | ~260 | Schema has `type: object` + `properties` | Missing `properties` key |
| **negative** | Error handling — invalid input | Empty object `{}` | PASS if execution fails OR no output (expected error) | Real API call (usually fast fail) | ~260 | Throws "Missing required field" | Returns success with empty input |
| **edge_case** | Boundary inputs | Heuristic (empty strings, very long values) | PASS if execution fails (expected) | Real API call | ~260 | Throws on null input | Returns 200 with garbage output |
| **dependency_health** | Upstream reachable | From `health_check_input` | Execution succeeds + basic validation | Real API call | ~260 | Gets response from upstream API | ETIMEDOUT |
| **piggyback** | Production correctness | Customer input (from /v1/do) | Schema validation against output_schema | FREE (piggybacks on customer call) | ~70 (growing) | Output matches schema | Missing expected field |

### 2.2 Test scheduling mechanics

**Implementation:** `setInterval`-based in-process scheduler (not cron, not pg-based)

```
Startup
  ├── 5s:   Browserless cache pre-warm (DB query)
  ├── 30s:  Initial scheduler sweep (runAdaptiveScheduler)
  ├── 45s:  Chromium probe (probeChromiumHealth)
  ├── 60s:  Dependency health checks
  ├── 5min: Weekly health sweep
  ├── 10min: Daily diagnostic
  ├── 2h:   Daily SQS snapshot
  ├── 3h:   Weekly data retention cleanup
  └── Recurring:
      ├── Every 1h:  Scheduler sweep
      ├── Every 6h:  Dependency health
      ├── Every 30m: Chromium probe
      ├── Every 24h: SQS snapshot + diagnostic
      ├── Every 7d:  Health sweep + retention cleanup
      └── Monday 8AM CET: Weekly digest email
```

**Adaptive interval per capability** (`computeAdaptiveInterval`, test-runner.ts:1159):
- Probation (<5 runs): min(6h, tier) — test aggressively
- SQS < 50: min(6h, tier) — intensify when degraded
- Normal: max(health_state_frequency, tier_minimum)
  - Health states: new/unstable=6h, recovering=12h, stable=24h, established=48h
  - Tier minimums: A=6h, B=24h, C=72h

**On process restart:** All timers reset. No catch-up logic. The initial sweep at 30s tests whatever is due based on `test_results.executed_at` timestamps. If the process was down for 6 hours, capabilities that were due during downtime will be tested in the first sweep.

**Maximum frequency:** 6 hours (Tier A + new/unstable health state)
**Minimum frequency:** 72 hours (Tier C + established health state, in practice capped at 48h by health state)

### 2.3 Failure classification

**7 verdicts** (failure-classifier.ts:151-238):

| Verdict | Confidence | Trigger | After-effect |
|---------|-----------|---------|-------------|
| **test_infrastructure** | HIGH | Missing env vars, quota exceeded, geo-restricted | Suppressed from SQS, triggers infra alert |
| **upstream_transient** | HIGH | HTTP 429/502/503/504, ECONNRESET, ETIMEDOUT, VIES errors | Tolerated in SQS (external service pattern), upstream tracker counts |
| **upstream_changed** | MEDIUM | Execution succeeded but validation failed + previously passed | Upstream tracker counts, potential escalation |
| **test_design** | MEDIUM | Execution succeeded but validation failed + never passed | No SQS penalty (test problem, not capability) |
| **capability_bug** | HIGH | TypeError, ReferenceError, SyntaxError, stack overflow | Full SQS penalty, lifecycle evaluation |
| **stale_input** | MEDIUM | Test input has expired dates | Auto-remediation candidate |
| **unknown** | LOW | None of above matched | Full SQS penalty, human review needed |

**Edge case risks:**
- A capability that causes an upstream to fail (e.g., sending malformed request) gets classified as `upstream_transient` — masking a real capability bug
- The `deterministic` reclassification (capability_type check) correctly catches this for pure-logic capabilities but misses it for `stable_api` types

**Production data (last 30 days):**
- 54,601 passed (no classification)
- 4,487 upstream_transient (6.9%)
- 4,375 unknown (6.7%) — **this is high; these need investigation**
- 1,003 test_infrastructure (1.5%)
- 488 capability_bug (0.7%)
- 447 upstream_changed (0.7%)
- 5 stale_input (<0.01%)

### 2.4 Test result validation logic

**Supported operators** (test-runner.ts:788-847):

| Operator | Logic | Used for |
|----------|-------|----------|
| `not_null` | `value != null` | Guaranteed field presence |
| `equals` | `value === expected` | Known-answer assertions |
| `contains` | `String(value).includes(expected)` | Substring matching |
| `in` | `values.includes(value)` | Enum validation |
| `type` | `typeof value === expected` | Type checking |
| `gt` / `lt` / `gte` | Numeric comparison | Range validation |
| `is_true` / `is_false` | Boolean check | Flag validation |

**Field reliability enforcement:**
- `guaranteed` → check enforced (failure = test fail)
- `common` → **skipped if field is missing** (acceptable absence)
- `rare` → skipped silently (never asserted)
- Without reliability annotations → all checks enforced (legacy behavior)

**Partial output:** A test passes with partial output as long as all `guaranteed` field checks pass. Missing `common`/`rare` fields do not cause failure.

### 2.5 Concurrency and ordering

- Tests within a capability run **sequentially** (for loop in `runTests`)
- Different capabilities can run in **parallel** only via event triggers (but the main scheduler is sequential)
- **No locks** — if two runs overlap for the same capability (scheduler + event trigger), both execute and both write results. The SQS uses a recency-weighted rolling window, so duplicate runs don't corrupt scoring — they just add more data points.
- Slow executors (30s+) don't block other capabilities — each capability's tests complete before the next starts in the scheduler sweep

### 2.6 Test data isolation

- **Test inputs are synthetic/known-good** — manifests contain real entity identifiers (e.g., Swedish org numbers, valid IBANs) but these are read-only lookups
- **No write operations** — capabilities are read-only (lookups, validations, extractions). No test can modify external state.
- **Rate limit risk:** Real API calls count against upstream rate limits. Tier A (6h) capabilities making 5 test types = 4 real calls per 6 hours. At 1,000 capabilities, this is ~660 calls/hour spread across many upstreams.
- **Browserless skip:** When Chromium is down, all ~50 scraping capability tests are skipped (prevents timeout pollution)

## Section 3: Self-Remediation — What Heals Automatically

### 3.1 Automated remediation actions

| Action | Trigger | What happens | Logged as | Recovery |
|--------|---------|-------------|-----------|----------|
| **Circuit breaker open** | 3 consecutive execution failures | Capability blocked from execution + backoff timer | lifecycle_transition | Half-open after backoff (5-30min exponential) |
| **Circuit breaker close** | Successful execution in half-open | Resets consecutive failure counter | lifecycle_transition | Immediate |
| **Test quarantine** | Manual or weekly sweep (not automatic today) | Test excluded from SQS scoring | classification | Weekly sweep checks for recovery |
| **Test status → upstream_broken** | 5+ upstream_transient in 48h | Suite marked, scheduler skips | upstream_escalation | Auto-recovery: 2 passes in 48h → normal |
| **Stale date auto-fix** | stale_input classification | Date fields updated to current year | auto_fix (HIGH confidence) | Immediate — test re-run passes |
| **Dead URL auto-fix** | ECONNREFUSED/ENOTFOUND on test URL | URL replaced with known-good fallback | auto_fix (MEDIUM confidence) | Next test run uses new URL |
| **Field rename auto-fix** | upstream_changed + unambiguous rename | Validation rules updated to match new field name | auto_fix (MEDIUM confidence) | Next test run passes |
| **Lifecycle: probation → active** | SQS ≥ 50 + qualified (≥5 runs) | visible = true, serves customer traffic | lifecycle_transition | — |
| **Lifecycle: active → degraded** | SQS < 25 OR circuit breaker | quality_warning added to /v1/do response | lifecycle_transition | SQS recovery → active |
| **Lifecycle: degraded → suspended** | 7 consecutive days degraded | Removed from matching, invisible | lifecycle_transition | Manual intervention |
| **Freshness decay** | No test in 4+ intervals | SQS score penalized, trend → "stale" | (implicit in score) | Next test run restores score |
| **Retry on transient failure** | ECONNRESET, HTTP 502/503/504, etc. | One retry with 1-2s delay | [retry] log | Success on retry = no failure recorded |
| **Upstream dependency-change trigger** | Dependency toggles healthy↔unhealthy | Re-tests up to 10 affected capabilities | (event trigger) | Tests determine new state |
| **On-deploy trigger** | Server restart | Tests degraded/suspended caps + random sample | (event trigger) | Recovery detected immediately |

### 3.2 Human intervention required

| Scenario | Detection method | Detection time | Signal to human | Estimated MTTR |
|----------|-----------------|---------------|-----------------|---------------|
| **Broken executor code** | capability_bug classification | 6-24h (next test run) | Interrupt email (if mass failure), meta-monitoring | 1-4h |
| **Upstream API permanently changed** | upstream_changed classification, 3+ in 7d | 1-7 days | upstream_escalation event, weekly digest | 2-8h (update executor) |
| **New API key needed** | test_infrastructure classification | 6-24h | Credential health check log, interrupt if critical (Browserless/Anthropic) | 15min (add key) |
| **Schema drift** | LOW-confidence proposal in auto_remediation_log | Weekly sweep | Weekly digest, health_monitor_events | 1-2h |
| **Rate limit changes** | upstream_transient spike | Minutes to hours | Meta-monitoring infra alert if >5 caps affected | 30min-2h |
| **New capability has no fixtures** | Readiness check fails | Immediate (on pipeline run) | Pipeline output | 15-30min (write fixtures) |

### 3.3 Quarantine lifecycle

```
normal ──(5+ upstream_transient in 48h)──→ upstream_broken
                                              │
                              (2+ passes in 48h via weekly sweep)
                                              │
                                          normal (restored)

normal ──(manual/diagnostic)──→ quarantined
                                    │
                     (3+ consecutive passes in 7d via weekly sweep)
                                    │
                                normal (restored)
```

**Quarantine impact on SQS:** Quarantined test results are still recorded but their classification (`test_infrastructure`, `upstream_broken`) is used by the failure classifier to distinguish them from real failures. The SQS scoring itself reads all test results — quarantine affects scheduling, not scoring directly.

**False quarantine risk:** If a test is marked `upstream_broken` but the upstream is actually fine (our code caused the failure), the test gets deprioritized and the real bug is masked. The 2-pass recovery requirement (48h window) means it takes at minimum 2 successful runs to un-quarantine — which might never happen if the underlying bug persists.

### 3.4 Upstream escalation coordination

**Correlated failure detection:**
- `checkInfrastructureHealth()` (meta-monitoring.ts:135-204): Detects >5 caps failing with same classification → fires `mass_failure` interrupt email
- `triggerOnDependencyChange()` (event-triggers.ts:182-225): On dependency health toggle, re-tests up to 10 affected capabilities
- **Browserless-specific:** `chromium-health.ts` probes every 30min; when down, scheduler skips ALL scraping capabilities (prevents ~50 timeout failures per cycle)

**But:** The `DEPENDENCY_CAPABILITY_MAP` in `event-triggers.ts` (line 52) is **still a hardcoded list of ~50 Browserless slugs** — this was not made dynamic when `chromium-health.ts` was updated. This is a maintenance gap.

**Recovery:** When Browserless recovers (detected by next probe), the scheduler starts including scraping capabilities again. The `triggerOnDependencyChange` fires re-tests for the affected capabilities.

### 3.5 Meta-monitoring (does the system monitor itself?)

| Monitor | What it detects | How | Frequency |
|---------|----------------|-----|-----------|
| `checkNewFailures()` | Regressions (was passing, now failing) | Queries recent results vs 30d history | Post-test-run |
| `checkInfrastructureHealth()` | Correlated failures (>5 caps, same classification) | Groups recent failures by classification | Post-test-run |
| `checkStaleTests()` | Tests not run in >30 days | Queries max(executed_at) per cap | Weekly |
| `checkScoreWithoutEvidence()` | SQS > 0 with <5 test results in 30d | Cross-checks capabilities vs test_results | Weekly |
| `checkStuckScores()` | SQS unchanged for >14d despite new results | Compares updated_at vs max(executed_at) | Weekly |
| `checkImpossibleScores()` | SQS > 100 or < 0 | Range check | Weekly (CRITICAL) |
| `checkFreeTierHealth()` | Free-tier caps SQS < 90 or not usable | Checks 5 specific free-tier caps | Daily (CRITICAL) |
| Daily diagnostic | Self-healing infrastructure | 15 test cases + scoring integrity | Daily |

**What's NOT monitored:**
- **Scheduler itself crashing** — no external watchdog. If the Node process crashes, Railway restarts it, but there's no alerting for "scheduler hasn't run in X hours"
- **Database slow queries** — no query timeout monitoring; a slow DB could silently delay test runs
- **Memory pressure** — no monitoring for growing in-memory caches (SQS cache, Browserless slug cache, rate limiter maps)

## Section 4: Scaling Analysis — CTO Critique

### 4.1 — 1,000 capabilities, Tier B daily

**Test volume:** 1,000 caps × 4 real-execution tests × 1 run/day = 4,000 test executions/day
**Duration:** Average execution time by type:
- deterministic: 9ms × ~240 = 2.2s
- stable_api: 7,467ms × ~320 = 40min
- scraping: 6,506ms × ~190 = 21min
- ai_assisted: 1,021ms × ~250 = 4.3min
- **Total sequential time: ~65 minutes** per daily sweep
- At 1,000 caps (proportional): ~4.2 hours per sweep
- The 1-hour scheduler interval means sweeps could overlap — but the scheduler is sequential, so it queues

**Database load:**
- 4,000 INSERT to test_results/day = ~167/hour
- 4,000 SELECT for validation rules = ~167/hour
- Plus SQS computation queries per affected capability
- **Manageable** for PostgreSQL — current load is 3,850/day (65,406 in 17 days)

**Memory:** In-memory caches are bounded (5-min TTL). No unbounded growth. At 1,000 caps, the Browserless slug cache and SQS caches are still small.

### 4.2 — Upstream API format change

**Detection chain:**
1. Next test run (6-24h depending on tier/health state)
2. known_answer test fails on validation rules
3. Classified as `upstream_changed` (execution succeeded but validation failed)
4. 3+ upstream_changed in 7d → `upstream_escalation` health event
5. SQS score degrades over the rolling 10-run window (1-5 days depending on frequency)
6. If SQS drops below 25 → lifecycle transitions to `degraded`
7. Weekly digest includes the failing capability

**Impact scope:** One upstream API change typically affects 1-3 capabilities (e.g., one country registry). The `DEPENDENCY_CAPABILITY_MAP` could detect correlated failures if the upstream is in the map.

**Recalibration:** The `--backfill --discover --fix` pipeline flags can auto-correct high-confidence fixture mismatches. But this requires a human to run the command.

### 4.3 — 100 capabilities fail simultaneously (shared upstream down)

**Detection:**
1. `checkInfrastructureHealth()` fires post-test-run: detects >5 caps with same classification → `mass_failure` interrupt email
2. If it's Browserless: `probeChromiumHealth()` detects within 30 minutes and **skips all ~50 scraping caps** on next sweep
3. If it's VIES: `triggerOnDependencyChange()` fires when VIES health check fails → re-tests 3 VIES-dependent caps (capped at 10)

**Does it detect correlation?** Partially:
- Browserless: YES — Chromium health skip prevents 50 false failures
- VIES: YES — dependency health + event trigger
- Other upstreams (GLEIF, BRREG, etc.): Only if <5 caps each. The `checkInfrastructureHealth` threshold is >5 caps with same classification, so 3 GLEIF failures wouldn't trigger mass-failure alert.

**Circuit breaker impact:** Yes, 100 circuit breakers would trip (3 consecutive failures each). After the upstream recovers, half-open probes would recover them one at a time over 5-30 minutes.

### 4.4 — Semantically wrong but structurally valid data

Example: upstream returns `{valid: true}` for an invalid IBAN because they changed validation logic.

**Detection:** The known_answer test for `iban-validate` asserts `{field: "valid", operator: "is_true"}` on a known-valid IBAN. If the upstream started returning `valid: true` for EVERYTHING, this test would still PASS. The negative test sends empty input and expects failure — it might catch a regression if the upstream stopped validating altogether.

**Gap:** There is NO known-bad input test (i.e., "send an invalid IBAN and assert `valid: false`"). The negative test only sends empty input. A semantic regression where the upstream becomes more permissive would be invisible to the current test suite.

**Mitigation needed:** Add `known_bad` test type that asserts expected-failure outcomes on known-invalid inputs.

### 4.5 — Test runner crashes mid-run

**State of in-flight tests:** Each test result is inserted into `test_results` immediately after execution. If the process crashes between execution and INSERT, that single result is lost. No orphaned transactions — the test runner doesn't create DB transactions that span multiple operations.

**test_run_log:** Written at the END of a run, not the beginning. A crash mid-run means no run_log entry. On restart, the initial sweep at 30s detects due capabilities from their last `test_results.executed_at` and re-runs them. No gap.

**Partial state risk:** If `persistDualProfileScores` crashes mid-way through a batch, some capabilities get updated scores and others don't. This is self-correcting on the next run. No corruption risk.

- `capability_bug` (escalated): 488 (4.5%)

**Auto-resolved rate: ~51%** (upstream_transient + test_infrastructure are handled without human intervention)
**Needs human: ~49%** (dominated by `unknown` classification — the biggest improvement opportunity)

**The 0 auto_fixed count** means the stale_date/dead_url/field_rename remediation rules have not triggered on production data. These are forward-looking — they'll activate when those specific failure patterns appear.

### 5.5 Time to detect quality degradation

| Stage | Time from upstream break | Mechanism |
|-------|------------------------|-----------|
| (a) Test fails | 6-24h (Tier A-B) | Next scheduled test run |
| (b) SQS score changes | Same test run | `persistDualProfileScores` runs after each batch |
| (c) Alert fires | Same test run + seconds | `checkNewFailures` if regression, `checkInfrastructureHealth` if mass failure |
| (d) Capability suspended | 7+ days | Requires SQS < 25 for 7 consecutive days |

**Fastest path:** Tier A capability with event trigger → 30 minutes (dependency change detection → targeted re-test).
**Slowest path:** Tier C capability with no dependency mapping → 72 hours until next scheduled test.

**With freshness decay:** Even without a test run, a capability's SQS visibly degrades after 4× its schedule interval. At Tier B (24h), that's 4 days of no testing before score decay begins.

### 5.6 Platform reliability at scale

**Single points of failure:**

| Component | What breaks | Blast radius | Recovery |
|-----------|-------------|-------------|----------|
| Node.js process crash | All scheduled jobs stop | All 256 caps lose monitoring until restart | Railway auto-restart (~30s) |
| Railway outage | API and scheduler down | Customer traffic + monitoring both offline | Railway SLA (~99.9%) |
| PostgreSQL down | All reads/writes fail | Everything — API, testing, scoring | Railway DB auto-recovery |
| Browserless outage | Scraping capabilities fail | ~50 capabilities (19%) | Chromium health skips tests; recovery on next probe |
| Anthropic outage | AI-assisted capabilities fail | ~68 capabilities (26%) | No coordinated skip — each fails individually |

**Critical gap:** No Anthropic-specific health skip equivalent to the Browserless health system. The `DEPENDENCY_CAPABILITY_MAP` has an empty array for Anthropic (commented: "Too many — skip individual listing"). If Claude goes down, 68 capabilities will fail independently, trip 68 circuit breakers, and generate a mass-failure alert.

### 5.7 External provider readiness

**Current state:** The platform assumes founder-as-sole-provider. For third-party providers at 1,000+ caps:

| Requirement | Current state | Gap |
|-------------|--------------|-----|
| Provider authentication | Not implemented | Need provider accounts, API key scoping |
| Sandboxed executor execution | Not implemented (direct code execution) | Need container isolation or WASM sandbox |
| Provider-submitted manifests | Pipeline supports YAML, but no validation beyond schema | Need provider portal, review queue |
| SLA enforcement | Lifecycle state machine exists | Need provider-specific SQS thresholds, suspension rules |
| Revenue sharing | Not implemented | Need provider billing, payout system |
| Trust verification | No provider identity verification | Need business verification, capability attestation |
| Rate limiting per provider | Only per-API-key | Need per-provider quotas |

## Section 6: Previously Undiscovered Systems

### 6.1 LLM-assisted test generation

**Not found.** Test generation is template-based (manifest → fixed patterns). No LLM involvement in creating test fixtures or validation rules. The `--discover` flag uses live execution output, not LLM analysis.

### 6.2 Fixture management (test_mode: live/fixture/canary)

**From DB:** 1,355 live, 10 fixture, 3 canary.
- `live` — calls real API every time
- `fixture` — uses saved baseline data (€0 external cost). `fixture_last_refreshed` tracks staleness.
- `canary` — periodic live check at reduced frequency

**Currently underutilized:** Only 10 fixtures and 3 canaries across 1,368 suites. Most tests are live.

### 6.3 Canary testing

3 canary suites exist. These run at reduced frequency as "is the API still alive?" probes. The dependency_health test type serves a similar role.

### 6.4 Event trigger system

Three triggers (event-triggers.ts): on-failure (regression verification), on-dependency-change (targeted re-test), on-deploy (spot-check). Rate-limited to prevent storms. **Still has hardcoded `DEPENDENCY_CAPABILITY_MAP`** — not yet made dynamic.

### 6.5 Interrupt email logic

5 interrupt types: suspension_warning, mass_failure, validation_failure, billing_alert, infrastructure_down. Deduplication: 1 email per slug+type per 24h. Uses Resend API. **4 interrupt emails sent in 17 days** — low volume, which is healthy.

### 6.6 Lifecycle state machine

Full implementation in lifecycle.ts (325 lines):
```
draft → validating → probation → active ⇄ degraded → suspended
```
Auto-transitions: probation→active (SQS≥50), active→degraded (SQS<25), degraded→active (SQS≥50 for 3 evals), degraded→suspended (7d). Anti-flap: 3 consecutive good evaluations required for degraded→active.

### 6.7 Upstream escalation coordination

`upstream-tracker.ts`: 5+ transient in 48h → mark suites as `upstream_broken`. 3+ changed in 7d → log for human review. Recovery: 2 passes in 48h → restore to normal.

### 6.8 Recalibration endpoint

`POST /v1/internal/tests/recalibrate` (internal-tests.ts) — re-runs a capability's tests with fresh input resolution, updates fixtures and baselines. Used for manual recalibration after upstream changes.

### 6.9 auto_remediation_log field

Written by `applyRemediation()` in auto-remediation.ts. Stores a JSON array of all remediation actions taken on a test suite, including: rule name, confidence level, what was changed, timestamp, whether it was applied or just proposed. **Currently empty for all suites** (0 auto_fixed in production).

### 6.10 failure_classification values

7 values: `upstream_transient`, `upstream_changed`, `upstream_degraded` (not seen in production), `test_infrastructure`, `test_design`, `capability_bug`, `stale_input`, `unknown`. Plus null for passed tests.

## Prioritized Lists

### CTO Reliability Concerns (things that would make a CTO hesitate)

1. **40.5% of failures are "unknown"** — 4,375 test failures in 30 days have no actionable classification. This is the biggest signal-to-noise problem. An agent calling the platform cannot distinguish "capability is broken" from "we don't know what happened."

2. **No known-bad input tests** — The test suite cannot detect semantic regressions where an upstream becomes more permissive (returns success for invalid input). A known_bad test type is needed.

3. **No Anthropic health skip** — If Claude goes down, 68 capabilities (26%) fail independently with no coordinated response. Unlike Browserless which has a dedicated health probe and test-skip mechanism, Anthropic failures create a storm of 68 circuit breaker trips.

4. **In-process scheduler with no watchdog** — All monitoring runs inside the same Node process. If it crashes between Railway auto-restarts, there's a monitoring gap. No external health check verifies "the scheduler actually ran in the last 2 hours."

5. **Probation capabilities can serve traffic without SQS** — A capability in `probation` state can be called by slug without any quality score. The platform-wide SQS floor of 25 is advisory only, not enforced on execution.

6. **0% auto-fix rate** — The remediation system has never successfully auto-fixed a test in production. The rules exist but haven't triggered. This could mean the rules are well-targeted (they handle future patterns), or it could mean they have dead-code bugs.

### VC Scaling Concerns (things that would make an investor question unit economics)

1. **Sequential onboarding pipeline** — Adding 1,000 capabilities requires running the script 1,000 times sequentially. No batch mode, no parallel processing. At 5 min/cap, that's 83 hours of pipeline time.

2. **Linear testing cost** — Monthly testing cost grows linearly with capability count. No optimization to reduce cost per capability (e.g., fixture caching, shared upstream probes, test deduplication for similar capabilities).

3. **Hardcoded dependency maps remain** — `event-triggers.ts` still has a ~50-slug hardcoded Browserless list. Every new scraping capability needs a code change to be included in coordinated dependency responses.

4. **No provider infrastructure** — Opening to external providers requires: sandbox execution, provider identity, review queues, revenue sharing, per-provider SQS thresholds. None of this exists.

5. **Fixture testing is underutilized** — Only 10/1,368 test suites use fixtures (0.7%). Moving more tests to fixtures would reduce external API costs by 60-80% while maintaining schema validation.

6. **6.7% unknown failure rate** — Nearly 1 in 15 test results provides no actionable signal. At 10,000 capabilities, this becomes ~2,900 unclassified failures per day that someone needs to investigate.

### Quick Wins (<1 day each, high impact)

1. **Make `event-triggers.ts` DEPENDENCY_CAPABILITY_MAP dynamic** — Same pattern as `chromium-health.ts`: query `capability_type = 'scraping'` from DB. ~30 minutes of work. Eliminates the last hardcoded capability list.

2. **Add known_bad test type to onboarding pipeline** — For each known_answer test, generate a complementary test with invalid input asserting failure/false. Detects semantic regressions. ~2 hours.

3. **Reduce "unknown" classification rate** — Analyze the 4,375 unknown failures to identify 2-3 new patterns for the failure classifier. Each new pattern moves hundreds of results from "unknown" to actionable. ~4 hours.

4. **Add Anthropic health probe** — Same pattern as Browserless: lightweight `/v1/models` check, shared health state, test-skip on failure. ~2 hours.

5. **Enforce platform SQS floor on execution** — Add a platform-level `min_sqs = 25` check in `matchCapability()` for non-probation capabilities. Prevents serving traffic from capabilities with known-bad quality. ~30 minutes.

6. **Add scheduler heartbeat** — Write a timestamp to `health_monitor_events` on each scheduler sweep. Add a meta-monitor that alerts if no sweep in 2 hours. ~1 hour.

7. **Expand fixture testing** — For deterministic capabilities (61 caps), convert all tests to fixture mode. For stable_api capabilities, convert schema_check tests (already dry-run). Reduces external API calls by ~15%. ~4 hours.
