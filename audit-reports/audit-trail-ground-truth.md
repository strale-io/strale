## 1. Data Model

All audit data lives in the `transactions` table (`apps/api/src/db/schema.ts:164-211`). No separate audit tables exist.

### Audit-relevant columns:

| Column | Type | Nullable | Purpose |
|--------|------|----------|---------|
| `auditTrail` | jsonb | YES | Full compliance record ŌĆö EU AI Act metadata, provenance, compliance mappings |
| `provenance` | jsonb | YES | Data source + fetch timestamp |
| `integrityHash` | varchar(128) | YES | SHA-256 hash of this record + previous hash |
| `previousHash` | varchar(128) | YES | Hash of prior transaction in chain |
| `transparencyMarker` | varchar(20) | NO, default 'ai_generated' | 'ai_generated', 'algorithmic', 'hybrid' |
| `dataJurisdiction` | varchar(10) | NO, default 'EU' | ISO 3166-1 region |
| `legalHold` | boolean | NO, default false | Retention hold for litigation |
| `isFreeTier` | boolean | NO | Whether audit record is publicly accessible |

FKs: `userId` ŌåÆ users, `capabilityId` ŌåÆ capabilities (nullable for solution rows), `solutionSlug` (nullable, for solution rows).

## 2. Hash Chain

**File:** `apps/api/src/lib/integrity-hash.ts` (94 lines)

### How it works:
- **Algorithm:** SHA-256
- **Genesis hash:** `sha256("strale-genesis-v1")`
- **Inputs to hash:** id, userId, status, input, output, error, priceCents, latencyMs, provenance, auditTrail, transparencyMarker, dataJurisdiction, createdAt, completedAt, previousHash
- **Chaining:** `getPreviousHash()` queries the most recent transaction by `completedAt` with a non-null `integrityHash` and returns its hash. New transactions link to this.

### Does it actually chain?
**Yes, but with caveats:**
- Hash is computed and stored **fire-and-forget** (`storeIntegrityHash(txnRecord.id).catch(() => {})` at do.ts:979). If the DB write fails, the hash is silently lost.
- Chain ordering relies on `completedAt` timestamp. Concurrent completions could produce non-deterministic ordering.
- The chain is per-day with per-transaction linking. Previous hash lookup uses `ORDER BY completedAt DESC LIMIT 1`.

### Tamper detection:
**`GET /v1/verify/:transactionId`** (`apps/api/src/routes/verify.ts:25-114`) walks the chain backward:
1. Fetches target transaction
2. Recomputes hash with stored `previousHash`
3. Compares against stored `integrityHash`
4. Walks backward through `previousHash` links, verifying each
5. Reports: `verified`, `chain_length`, `broken_links`, `reaches_genesis`

If you tampered with a row by hand, the next `GET /v1/verify/:id` call would detect it ŌĆö `hash_valid: false` for that row and `broken_links > 0` in the chain.

### Populated for ALL executions?
Yes ŌĆö both success and failure paths construct audit trails. `buildFailureAudit()` (do.ts:1915-1946) covers failure cases with a subset of fields.

**Gap:** Solution execution audit trails (`solution-execute.ts:350-375`) are constructed inline with a simpler shape ŌĆö no `data_source_url`, no `input_hash`, no `compliance.regulations_addressed`. They have per-step breakdowns instead.

## 4. API Surface

### Endpoints returning audit data:

| Endpoint | Auth | Returns |
|----------|------|---------|
| `GET /v1/transactions/:id` | API key (own txns) or free-tier by ID | Full row including `audit_trail`, `provenance`, `integrity_hash` |
| `GET /v1/transactions/:id/verify` | Same as above | Hash verification + chain walk results |
| `GET /v1/audit/:transactionId?token=...` | HMAC token (no API key needed) | Audit record + compliance data |
| `GET /v1/verify/:transactionId?depth=N` | Public (rate limited) | Chain verification with configurable depth |

### In execution responses:
The `POST /v1/do` response includes audit data inside `meta.audit` for every execution. Callers get the audit trail automatically ŌĆö no separate request needed.

### Shareable URL:
Generated by `getShareableUrl()` in `audit-token.ts:14-17`. Format: `https://strale.dev/audit/{id}?token={hmac}`. Token is 32-char HMAC-SHA256 substring. Included in every `audit_trail.compliance.shareable_url`.

### SDK helpers:
None. Both TypeScript and Python SDKs type `audit` as `Record<string, unknown>` / `Dict[str, Any]` ŌĆö no typed helpers for consuming audit records.

## 7. Honest Assessment

If a prospective customer asked "show me the audit trail for a past call," you could:

1. **Give them the shareable URL** ŌĆö `https://strale.dev/audit/{id}?token={token}` is generated for every execution and included in the API response. However, the strale.dev page currently renders mock data, not real data. The backend endpoint (`/v1/audit/:id`) returns the real audit record correctly ŌĆö a developer could consume it via curl. A non-technical auditor clicking the link would see a polished UI with fake data.

2. **Show them the hash chain verification** ŌĆö `GET /v1/verify/:id` returns chain length, broken links, and whether it reaches genesis. This is fully functional and publicly accessible.

3. **Show them the raw transaction record** ŌĆö `GET /v1/transactions/:id` returns the full `audit_trail` JSONB with EU AI Act article mappings, provenance, input hash, and compliance metadata. This is the most complete view.

**The weakest link** is the frontend. The backend audit trail is comprehensive and cryptographically verifiable. The frontend page that renders it for non-technical users is wired to mock data. A customer clicking the shareable URL sees a beautiful compliance page ŌĆö with fake data. Connecting the frontend to the real API is a one-session fix (fetch from `/v1/audit/:id?token=...` instead of mock constants), but it hasn't been done.

**Second weakness:** the fire-and-forget hash storage pattern. If the database write after `storeIntegrityHash()` fails, the chain breaks silently. In practice this hasn't happened (invariant probes show 0 broken links), but it's architecturally fragile.

**Third weakness:** solution execution audit trails are simpler than capability audit trails ŌĆö no EU AI Act article mappings, no `input_hash`, no `data_source_url`. This is documented as a TODO in `solution-execute.ts:272`.
