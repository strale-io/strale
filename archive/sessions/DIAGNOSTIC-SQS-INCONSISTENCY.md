# Diagnostic Report: SQS Score Inconsistency Between Solution and Capability Pages

**Date:** 2026-03-24
**Status:** Diagnostic only — no code changes made

---

## Executive Summary

The same capability shows different SQS scores depending on whether you view it on its own capability detail page vs. inside a solution detail page. The root cause is that **two different data paths** serve the same conceptual data:

- **Capability detail page** → live-computes `computeDualProfileSQS()` at request time
- **Solution detail page** → reads pre-computed `matrix_sqs` from DB columns

These diverge when test results arrive between persist cycles, or when freshness decay accumulates at different rates.

---

## Concrete Example

`tech-stack-detect` capability:
- **Solution page** (`/solutions/token-project-dd`): SQS **59**, Quality: **B**, Reliability: **C**
- **Capability page** (`/capabilities/tech-stack-detect`): SQS **75**, Quality: **B**, Reliability: **B**, Trend: **Improving**

## Step 6: Staleness/Freshness Logic

### Freshness Decay Computation
- **File:** `apps/api/src/lib/freshness-decay.ts`
- Tier-based: A=6h, B=24h, C=72h
- Levels: fresh (0-2x), aging (2-4x), stale (4-8x), expired (8-12x), unverified (>12x or >30d)
- Decay: 1 point per interval past the 4th

### Staleness Refresh Job
- **File:** `apps/api/src/jobs/refresh-stale-scores.ts`
- Runs every 2 hours
- Re-applies freshness decay to `matrix_sqs` for capabilities with stale `freshnessDecayedAt`
- **Covers capabilities ONLY — no solution-level refresh**

### Invariant Checker
- **File:** `apps/api/src/jobs/invariant-checker.ts`
- Runs every 2 hours + 60s after startup
- CHECK 1: Score persistence drift (auto-heals via `persistDualProfileScores()`)
- CHECK 2: Solution score sanity (alert only)
- CHECK 3: Orphaned solution steps (alert only)
- CHECK 4: Freshness decay drift (auto-heals)

- **File:** `strale-frontend/src/lib/trust-display.ts:158-168`

## Step 11: All Root Causes

### RC-1: Live computation vs DB read divergence (PRIMARY)

**The capability detail trust endpoint computes SQS live; the solution detail trust endpoint reads it from DB.**

- `internal-trust.ts:487` — `computeDualProfileSQS(slug)` runs at request time for capability page
- `internal-trust.ts:792` — `step.matrixSqs` reads from DB for solution page

Between `persistDualProfileScores()` calls (which run after test completions + every 2h by staleness refresh), new test results accumulate. The live computation sees the latest data; the DB columns are a snapshot.

**Impact:** The capability page shows a score based on the latest test results (potentially improved). The solution page shows an older score. In the example, tech-stack-detect likely had recent tests that improved its score to 75, but the DB still holds the older decayed value of 59.

### RC-2: Freshness decay applied at different timestamps

- `persistDualProfileScores()` (test-runner.ts:1875-1882) applies freshness decay at persist time
- The capability detail endpoint (internal-trust.ts:490-495) applies freshness decay at request time
- The solution detail endpoint reads the already-decayed `matrix_sqs` from DB

If the DB was persisted hours ago (when the capability was more stale), the DB `matrix_sqs` has MORE decay than the live computation (which may see a recent test as "fresh").

**Impact:** This compounds RC-1. A test running after the last persist resets freshness to "fresh" for live computation (0 decay) but the DB still holds the previous decayed value.

### RC-3: QP/RP scores also diverge between live and DB

- Capability page: live QP grade from `dual.qp.grade` (based on current test results)
- Solution page: DB `qp_score` → `gradeFromScore()` (snapshot)

In the example: Reliability shows "B" on capability page (live RP score >= 75) but "C" on solution page (DB `rp_score` was 50-74 at persist time).

### RC-4: Solution detail response omits freshness_level

- The solution list endpoint (`solutions.ts:74-116`) returns `freshness_level` and `last_tested_at`
- The solution detail trust endpoint (`internal-trust.ts:859-907`) does NOT return these fields
- The frontend must infer staleness solely from the `trend` field ("stale")

**Impact:** The frontend cannot display accurate freshness information on the solution detail page.

### RC-5: "Reliable for production use" ignores freshness/trend

- `getVerdict(score)` in `ZoneBReliability.tsx:45-50` only checks SQS score
- Score >= 75 → "Reliable for production use" regardless of trend being "stale"
- Both messages appear simultaneously, contradicting each other

**Impact:** Users see "Reliable for production use" AND "Stale — not recently tested" on the same page.

### RC-6: Solution-level freshness not aggregated in detail trust endpoint

The solution detail trust response (lines 859-907) computes `solutionTrend` but does NOT compute or return:
- `freshness_level` (computed by `worstFreshnessLevel()` in trust-labels.ts, but not called here)
- `last_tested_at` (computed by `oldestTestedAt()` in trust-labels.ts, but not called here)

The solution LIST endpoint does compute these, creating an inconsistency between list and detail views.

### RC-7: Cache layering amplifies divergence

Three independent caches:
1. `dualCache` in `sqs.ts:663` — 10-minute TTL for `computeDualProfileSQS()` results
2. SWR cache in `internal-trust.ts:41-42` — 2-min fresh, 30-min stale for endpoint responses
3. DB columns — updated only by `persistDualProfileScores()` (after test runs + every 2h)

The worst case: capability page serves a 10-min-cached live computation that is 10 min stale. The solution page serves a 30-min SWR-cached response built from DB columns that are up to 2h stale. Total divergence window: up to ~2.5 hours.

## Step 12: Fix Plan

### Fix 1: Make capability detail endpoint read from DB (like everything else)
- **File:** `apps/api/src/routes/internal-trust.ts:436-598`
- **Change:** Replace `computeDualProfileSQS(slug)` call (line 487) with a DB read of `matrixSqs`, `matrixSqsRaw`, `qpScore`, `rpScore`, `trend`, `freshnessLevel`, `lastTestedAt` from the capabilities table
- **Retain:** Live computation ONLY in the `/v1/quality/:slug` public endpoint (which is explicitly designed as "compute on demand" with detailed factor breakdowns)
- **For the factor breakdown** (QP/RP factors detail): Either read from a cached computation or accept that the detail view calls `computeDualProfileSQS()` but uses DB `matrixSqs` as the headline score
- **Expected outcome:** Capability page and solution page show identical per-capability scores
- **Repo:** strale (backend)

### Fix 2: Add freshness_level and last_tested_at to solution detail trust response
- **File:** `apps/api/src/routes/internal-trust.ts:859-907`

- **Prerequisite:** The per-step data (lines 791-807) must also read `freshnessLevel` and `lastTestedAt` from the capabilities JOIN (already available in the query at lines 776-777 but not passed through to stepData)
- **Expected outcome:** Solution detail page has accurate freshness information
- **Repo:** strale (backend)

### Fix 3: Make getVerdict() freshness-aware
- **File:** `strale-frontend/src/components/solutions/ZoneBReliability.tsx:45-50`
- **Change:** Pass trend/freshness to `getVerdict()`. If trend === "stale" or freshness_level is "stale"/"expired"/"unverified", return "Quality data may be outdated — not recently tested" instead of "Reliable for production use"
- **Expected outcome:** "Reliable for production use" and "Stale" warnings never appear simultaneously
- **Repo:** strale-frontend

### Fix 4: Include freshness_level in per-step data on solution trust endpoints
- **File:** `apps/api/src/routes/internal-trust.ts` — both `/solutions/batch` (line 683-699) and `/solutions/:slug` (line 791-807)
- **Change:** Add `freshness_level` and `last_tested_at` to the per-step return objects (the data is already in the query, just not mapped)
- **Expected outcome:** Frontend can display per-step freshness indicators in the solution quality breakdown
- **Repo:** strale (backend)

### Fix 5: Reduce persist lag (optional, lower priority)
- **File:** `apps/api/src/lib/test-runner.ts` (the `runTests()` caller at line 294)
- **Change:** Ensure `persistDualProfileScores()` is called immediately after every test batch completes (it already is), and consider adding a more frequent scheduled re-persist (e.g., every 30 min instead of relying on the 2h invariant checker)
- **Expected outcome:** DB columns stay closer to live computation, reducing divergence window
- **Repo:** strale (backend)
